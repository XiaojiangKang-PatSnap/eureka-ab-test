SECTION 01

Executive summary

Eureka Life Sciences sells one product to at least six distinguishable buyers — computational chemists, BD leaders, IP strategists, academic PIs, data analysts, and time-poor evaluators. The production page at eureka.patsnap.com/ls-landing serves all of them with a consensus message that performs adequately for the broad market but compromises depth for any one segment.

The ten variants in this repository are not stylistic alternatives. Each one is an experiment in specializing the message for a specific persona, traffic source, and stage in the buyer journey — while remaining inside PatSnap's design system so that visual consistency is preserved and the only thing that varies is content, structure, and information density.

This report explains, for each variant: who it is for, why it exists, the user scenarios where it wins, where to deploy it (ad creative, email campaign, search intent, traffic source), and where it underperforms. It closes with a concrete recommendation for how to run the A/B test so the results are interpretable rather than diluted.

The headline recommendation: do not split traffic ten ways. Either bucket users by source first and run small A/B tests inside each bucket, or pick two or three variants that move only one variable versus production and learn causal answers. A flat ten-way split will produce noisy data that cannot be acted on.

SECTION 02

Context — what the production page actually sells

Before evaluating variants, it is worth being precise about the product, the buyers, and the jobs-to-be-done that the production page is trying to support. Everything in section 5 is graded against this baseline.

Product surface

Eureka Life Sciences is an agentic AI suite for early drug discovery. The platform exposes six task-specific agents, each addressing a discrete workflow in biopharma R&D:

  • Lead Compound Analysis — score and rank candidates against ADMET, drug-likeness, and PK profiles; compare structures against patent and literature evidence.
  • SAR Data Extraction — pull structure-activity relationships from millions of patents and papers; export to FASTA, SDF, and SMILES.
  • Pulse Briefs — weekly intelligence on priority targets covering filings, clinical readouts, and competitor moves.
  • Document Analyzer — parse 100+ page patents and trial reports with citation-grade traceability.
  • Target Intelligence — map every program targeting a pathway, broken down by modality, geography, developer, and phase.
  • Clinical Benchmark — side-by-side comparison of trials across endpoints, populations, and dosing schedules.

Pricing and access

Two paid tiers plus an API: Pro at roughly $400 per user per month, Enterprise with custom pricing, and a developer-facing REST + Python SDK. This pricing model implies the page must do double duty — converting individual researchers on Pro and qualifying enterprise leads for sales.

Data scale claims

The production page leans hard on scale: 1.2B+ biological sequences, 160M+ chemical structures, billions of patents, papers, and clinical trial records. These claims function as a credibility anchor for the AI agents above them — without that scale, the agents would be just another wrapper.

Six personas, one page

The production page tries to address six distinguishable buyers in a single scroll:

  • Computational and medicinal chemists — care about SAR, ADMET, and chemical export formats. Trust scientific specificity over marketing language.
  • BD and competitive intelligence leads — care about pipeline mapping, deal sourcing, and weekly briefs. Trust customer logos and named case studies.
  • IP and FTO professionals — care about patent landscape analysis and citation traceability. Trust audit trails and compliance language.
  • VP-level R&D buyers — care about ROI, throughput, and de-risking programs. Trust quantified outcomes and customer references.
  • Academic researchers and KOLs — care about peer-reviewed validation and methodology. Trust journal names and citation formats.
  • Data scientists and analysts — care about API access, data freshness, and integration paths. Trust working code samples and uptime numbers.

One page cannot land equally well for all six. The variants attempt to fix that by specializing — but each one only earns its place if it is paired with traffic that matches its persona.

Jobs-to-be-done

The production page references five canonical workflows in its case study tabs: Target Assessment, Lead Optimization, Clinical Benchmark, IP Strategy, and Competitive Intelligence. These map cleanly onto the six personas above and serve as a useful checklist when evaluating which variant covers which job.

SECTION 03

Why ten variants exist (the testing thesis)

The variants are not a brand exploration exercise. They exist because the production page makes three structural compromises that are visible once you read it from each persona's point of view.

Compromise 1 — Hero composition

The production hero leads with a brand statement ("Make every biopharma decision count") and a generic CTA. This is safe but it does not signal which audience the page is for. A computational chemist scanning above the fold sees marketing language; a BD lead sees the same marketing language. Neither feels recognized.

Compromise 2 — Section ordering

Case studies, agent showcases, customer logos, testimonials, and FAQ all appear in a fixed sequence. For an enterprise buyer evaluating rollout, the case studies should arrive earlier; for a developer evaluating integration, the API surface should appear before the customer logos. The production page cannot do both simultaneously.

Compromise 3 — Information density

The production page averages a pace appropriate for a generalist reader. A bench scientist who already knows what SAR and ADMET mean will find it light; a non-technical procurement reader will find it dense. Either way, somebody is paying a tax.

Three meta-axes the variants move along

Every variant in the set differentiates along at least one of three axes. Understanding these axes is the key to picking a variant for a given traffic source.

Axis 1 — Depth

Variants A, F, H, and J move toward technical depth — SAR distributions, code samples, dashboards, peer-reviewed citations. Variants D, E, and G move toward business depth — ROI metrics, pricing transparency, customer outcomes. Variants B, C, and I move toward brand and positioning depth — modern biotech aspiration, scholarly tone, mobile fluency.

Axis 2 — Proof

Variants prove the product through different evidence types. A and H prove via product surfaces (charts, ADMET tables). G proves via case studies and named customers. C and J prove via peer-reviewed publications. D proves via quantified ROI. E proves via transparent pricing. F proves via working code.

Axis 3 — Surface

Most variants are light-mode. A is dark-mode (matching how computational chemists work in their IDEs). I is mobile-first. The surface choice is a signal — dark says "engineering tool," mobile-first says "for the move."

What the variants do not do

Every variant in this set ships against PatSnap's design system: same brand color, same typography, same component primitives. They differ in content choice, section order, and information density — never in visual identity. This matters because it means the test measures message-market fit, not aesthetic preference. Aesthetic A/B tests are usually unactionable; message A/B tests are.

Designer note — if visual variation feels too tame, that is the point. The first iteration of these pages used different fonts, gradients, and color palettes per variant. Bringing them all under the design system means we can attribute lift to message and structure rather than to whether the user happened to like a purple gradient.

How to read this table

The table is intentionally opinionated. The "best-fit traffic source" column is what makes a variant earn its keep — a variant served to the wrong audience will underperform the production page on every metric. The discipline this report argues for is to match each variant to a single traffic source rather than letting the test split traffic indiscriminately.

The success metric column matters because it changes per variant. A trial signup is the right metric for B and E; an enterprise demo booking is the right metric for D and G; an API key issued is the right metric for F. Optimizing every variant against a single conversion event would punish variants whose job is to qualify, educate, or hand off.

SECTION 05

Version-by-version analysis

Each profile below covers persona portrait, why the variant exists, user scenarios, content rationale, deployment guidance, and failure modes. Profiles are ordered A through J to match the file naming.

A
Version A — Dark Tech
Computational chemist · dark surface · SAR + ADMET hero
Dark mode Technical

Persona portrait

The reader is a computational or medicinal chemist with five to fifteen years of experience. Their day is split between Schrödinger or RDKit notebooks, internal SAR review meetings, and conversations with bench chemists about which compounds to push next. They are deeply technical but not necessarily senior — many are individual contributors whose recommendations carry weight because of demonstrated rigor, not because of title. They live in dark-mode IDEs and dark-mode Slack, and they have a finely tuned aversion to anything that looks like marketing for a tool they will actually use.

What they fear: tools that overstate accuracy, tools that hide the underlying data, and tools that produce numbers they cannot defend in front of a more senior chemist. What they trust: precise units, visible confidence intervals, ADMET parameters they recognize from the literature (LogP, TPSA, hERG, CYP3A4), and the ability to export to formats that match their existing workflow (SDF, SMILES, FASTA). They are skeptical by default and convert when they see specificity, not enthusiasm.

Why this variant exists

The production page mentions SAR and ADMET in passing but does not lead with them. A computational chemist scanning above the fold sees "Make every biopharma decision count" — a brand statement that does not signal whether the underlying tool understands their workflow. Version A fixes that by making the hero a working SAR distribution paired with an ADMET grid, using mono-formatted numerics and a dark surface that signals "this was built by people who do this work."

The compromise the production page makes — generic above the fold, specifics below — costs traffic from this persona because they decide whether to keep scrolling in the first three seconds. Version A wins those three seconds.

User scenarios

Scenario 1 — A medicinal chemist at a 50-person biotech sees a tweet linking to "an SAR extraction tool that exports to SDF." She clicks expecting either a wrapper or a real tool. The dark surface and the SAR distribution chart in the hero tell her within five seconds that this is a real tool. She reads the ADMET grid (LogP 2.84, hERG 0.18 — values she recognizes as plausible), scrolls to the agent grid, and starts the free trial.
Scenario 2 — A senior computational chemist is evaluating tools for his team and has been sent links to four competitors by his director. Three of the competitor pages lead with founder photos or marketing language. Version A leads with a chart and a compliance row (FDA 21 CFR Part 11, SOC 2). He forwards Version A to his procurement contact as the technically credible candidate.
Scenario 3 — A bench chemist who works with the computational team gets a Slack message from a colleague: "you should look at this." She is skeptical of AI tools — most have promised more than they delivered. The dark surface puts her at ease (it doesn't read as marketing), and the agent grid showing six task-specific tools rather than a single "ask anything" chatbot makes her willing to start a trial.

Content rationale

The hero composition is the most important decision in this variant. The left column carries the headline, three mono-formatted scale stats (1.2B+ sequences, 160M+ structures, 99.2% accuracy), and two CTAs. The right column carries a working SAR distribution bar chart with mono-formatted axis labels (10nM, 100nM, 1µM, 10µM) and an ADMET parameter grid below it with semantic coloring (green for pass, orange for caution). This is a deliberate choice to put a product surface in the hero rather than a stock illustration — the chart is the credibility argument.

The agent grid uses six cards rather than four to fully cover the workflow surface: Lead Compound, SAR Extraction, Pulse Briefs, Document Analyzer, Target Intelligence, Clinical Benchmark. A computational chemist reading this should recognize all six as real workflows they touch in any given week. Cutting to four would force a choice that pushes some of them away.

The compliance row near the bottom (FDA 21 CFR Part 11, ISO 27001, SOC 2 Type II, GDPR, RBAC, Audit Trail) is positioned for the moment when the chemist has decided they like the tool and starts thinking about how to get it through procurement. It is not a hero element — it is a "now I need to forward this to security" element, and putting it lower in the page is correct for the persona.

Where to deploy

Paid: Twitter / X ads targeting RDKit, OpenEye, and Schrödinger user lists; LinkedIn ads targeting "Computational Chemist" or "Medicinal Chemist" titles at biotech and CRO companies. Organic: guest posts on Practical Cheminformatics or the OpenEye blog, with the post linking to Version A rather than the production page. Email: outbound campaigns from sales reps targeting computational leads at named accounts. Communities: a link in scientific Slack workspaces, especially Bay-area-bio or Boston-bio channels.

Ad creative should lead with a chart, not a person — "See how Eureka extracts SAR from a single patent in 4 seconds" beats "Accelerate your discovery." The dark surface in the variant is consistent with dark-themed ad creative, which performs well on Twitter / X and dev-leaning LinkedIn audiences.

Where it underperforms

Version A is the wrong page for executive buyers, BD leaders, and any procurement contact who arrives without scientific context. The dark surface and the technical specificity will read as "this is for engineers, not for me" and the visitor will bounce. It is also the wrong page for a generic Google search like "drug discovery AI" — the searcher has not signaled they are technical, and the production page will outperform here.

It will also underperform on mobile despite being responsive. The SAR chart and ADMET grid degrade on small screens, and the persona this variant targets primarily browses on desktop. If mobile traffic to A is significant, route those visitors to Version I instead.

B
Version B — Modern Biotech
Young biotech R&D lead · live agent activity · ship-it culture
Modern Energetic

Persona portrait

The reader is a Director of Discovery, Head of Biology, or VP of Research at a series-A through series-C biotech with twenty to two hundred employees. Their team is small enough that they personally know every scientist, fast enough that decisions get made over Slack, and ambitious enough that they describe themselves as "ship-it culture." They have a runway of eighteen to thirty months and they are aware, every day, that they need to move faster than their better-funded competitors.

What they fear: tooling decisions that slow the team down, vendor lock-in, and onboarding processes that consume senior scientists' time. What they trust: peer biotechs that already use the tool, fast trial-to-value paths, and language that reflects how their team actually talks. They do not want to be sold to in the same register as a top-ten pharma — they want to be recognized as a different kind of buyer.

Why this variant exists

The production page is institutionally calibrated. It mentions "biopharma organizations" and lists global pharma logos, both of which read as "this is for the big guys" to a 75-person biotech founder. Version B trades the institutional register for a peer register: "supercharge your research," "ship insights your competitors can't match," "talk to founders" instead of "contact sales."

It also fixes a structural problem in the production page: the agents are described in the abstract rather than shown in motion. Version B's hero is a live activity stack — four agent rows with status tags ("Done," "Running," "New") that look like a notification feed. For a buyer who lives in Slack and Linear, this is immediately recognizable as "the tool fits how we work."

User scenarios

Scenario 1 — A 30-person biotech's Director of Discovery sees a LinkedIn ad: "Run patent and lit reviews in minutes, not weeks." She clicks. The hero shows agent activity that looks like her team's Slack: ACVR2A done, KRAS G12C running, GLP-1R weekly brief new. She recognizes the workflow shape and starts the trial without filling out a sales form.
Scenario 2 — A research scientist forwards a Substack post to her boss with the message "this looks useful." Her boss clicks the link in his lunch break, scans the page on his iPhone (it works), and forwards the link to two other team leads with "let's try this." Three of them sign up for trials by the end of the day. Team-invitation rate is the metric this variant should be judged on, not raw signups.
Scenario 3 — A founder is tired of manually checking patent feeds for competitor moves. He sees Version B's case study row mentioning "Ivonescimab vs Keytruda" — exactly the kind of head-to-head he ran last week, by hand, in eight hours. He starts the trial that night, runs the same comparison in twelve minutes, and forwards screenshots to his CEO.

Content rationale

The hero is the differentiator. Instead of a chart (Version A's choice) or a static illustration (the production choice), Version B's right column is a live agent activity stack — four rows showing different agents in different states, with semantic tags. The choice maps to how the persona thinks about software: their existing tools (Linear, Slack, Notion) all use this same activity-stream pattern. Showing the product in that idiom shortcuts the "is this for me" judgment.

The agent grid is unchanged from production in scope (six agents) but the copy is rewritten: "Try it" instead of "Learn more," "ship faster" instead of "accelerate decision-making." This is not a stylistic preference — it is a deliberate signal that the variant speaks the buyer's dialect.

The case study row is restricted to three concrete examples (ACVR2A, SIK inhibitor SAR, Ivonescimab vs Keytruda) rather than the production page's six. Three is enough to demonstrate range; six dilutes the message and makes the page longer than the persona will read on a phone or in a five-minute coffee break.

The CTA copy ("Try free for 14 days," "Talk to founders") is the most aggressive specialization in the variant. "Talk to founders" is a deliberate choice — it signals that this is a startup-to-startup conversation, not a procurement-to-vendor one. It will lose some leads who expect formal sales engagement, and that is acceptable: those leads are better served by Version D or G.

Where to deploy

Paid: LinkedIn ads targeting "Director of Discovery," "Head of Biology," and "VP of Research" titles at companies with 20–200 employees in the biotech industry. Organic: Endpoints, STAT, BiotechTV, and emerging-biotech newsletters; co-marketing posts with peer companies (e.g. with a CRO or a tools company that targets the same segment). Communities: Bay-area biotech Slack groups, Boston biotech meetup mailing lists, BIO conference QR codes. Email: founder-led outbound to specific named accounts.

Where it underperforms

Version B is the wrong page for top-10 pharma evaluators. The "ship-it" register reads as unserious to a procurement officer at a 100,000-person organization, and the lack of compliance and security framing will leave them with unanswered questions. Route this audience to Version D or G instead.

It is also a poor match for academic users. Postdocs and PIs do not respond to "talk to founders" the same way a startup-stage R&D lead does — they want methodology, peer-reviewed validation, and citation language. Route academic traffic to Version C or J.

C
Version C — Academic Rigor
Academic PI · centered hero · publications + methodology
Academic Validated

Persona portrait

The reader is an academic principal investigator, a senior postdoc running their own group within a larger lab, or a research-track scientist at a hospital or institute. They have published in journals their target audience reads, and they care about whether a tool can be cited as a methodology in their next paper. They are slower to adopt than industry counterparts because every tool they introduce has to clear an internal bar — does this preserve scientific rigor, can I defend the methodology to a reviewer, will the underlying logic be reproducible if a grad student inherits the project.

What they fear: tools that are not citable, tools whose methodology is opaque, and tools that disappear in three years leaving published work that depends on them stranded. What they trust: peer-reviewed publications, institutional partners (Harvard, MIT, Stanford), explicit methodology descriptions, and the existence of a citation block they can drop into a manuscript without rewriting.

Why this variant exists

The production page mentions Cleveland Clinic and St. Jude in its logo wall but does not lean into academic credibility. Version C exists because academic users — particularly those who might bring their institution as a paying customer — need to see methodology before they see CTAs. The variant inverts the production page's structure: the hero is a centered statement of validation ("Advancing drug discovery with rigorously validated AI"), and the second section is a numbered publication list with journal names and DOIs.

The structural change matters more than the copy change. By placing publications above the agent showcase, Version C signals "we are a research platform first, a product second." For an academic reader, this ordering is the difference between trusting the tool and dismissing it as a commercial wrapper.

User scenarios

Scenario 1 — A computational biology PI sees a citation in a recent Nature Biotechnology paper that mentions Eureka as the platform used for SAR extraction. Curious, they search for the platform and land on Version C. The hero references the same publication. They scroll to see methodology described in four explicit stages, then to the citations row, and bookmark the page to share with their two postdocs.
Scenario 2 — A senior postdoc is preparing a grant proposal that includes patent landscape analysis as a methodology. She needs a tool whose methodology she can describe in the methods section. Version C's stage-by-stage methodology section maps almost directly to a methods paragraph. She uses it as the basis for her grant text and submits a research access request.
Scenario 3 — A library science professional at a research university is evaluating which AI research platforms to add to the institution's licensing portfolio. They forward Version C to the chemistry and biology department heads with the message "this one is the credible one." The four publications and the partner-institution logos do most of the work in that internal forward.

Content rationale

The hero is centered rather than two-column because the persona reads more like a journal article than a SaaS landing page. Centering the hero — with publication names called out in the subtitle — is a deliberate echo of the typographic conventions of academic publishing. The variant uses the same Inter typeface as every other variant (per the design system), but the layout signals scholarly tone without reaching for a serif font.

The publication list is rendered as a numbered citation block, with mono-formatted reference numbers ([01], [02], [03], [04]) and DOI strings rendered in the brand color. Each entry includes title, authors, journal, and year. This is the format the reader uses every day. Imitating it builds trust faster than restating it as marketing copy.

The four-stage methodology section uses mono-formatted stage labels ("STAGE 01", "STAGE 02") and explicit stage descriptions: data ingestion, AI analysis, validation, knowledge graph. This is the closest thing on any variant to a methodology section in a paper, and it is positioned to be screenshot-able and copy-pasteable into a grant proposal or methods section.

The data scale section reads slightly differently than on other variants. The fourth stat is "14 peer-reviewed citations" rather than a generic count of customers — a small change that signals the variant's optimization target is academic legitimacy, not enterprise scale.

Where to deploy

Paid: ResearchGate display ads targeting members in pharmacology, medicinal chemistry, and computational biology fields. Organic: guest blog posts on academic computational chemistry blogs; conference QR codes at ACS, Cold Spring Harbor, and AACR meetings; mailing list co-mentions with academic-friendly tools. Email: outbound campaigns to postdoc Slack channels and institutional licensing contacts. Co-marketing: partnerships with publishers (Wiley, Elsevier) where Eureka's methodology is referenced in editorial content.

Where it underperforms

Version C is the wrong page for industry buyers who came in through paid sales channels. The publication-first ordering will read as slow and academic to a VP who needs to know whether a tool will hit ROI in the current fiscal year. It will also underperform with developer audiences — the methodology section explains how the platform is built but does not show the API surface, so a builder will leave looking for code samples and route themselves to Version F.

One subtle failure mode worth flagging: Version C is also a wrong choice for academic users who already have institutional access and just need to start using the tool. For them, the credibility argument is redundant and the page buries the actual product behind too many proof sections. Route returning academic users to Version E instead.

D
Version D — ROI / Business
VP R&D · ROI panel · quantified outcomes
Business Enterprise

Persona portrait

The reader holds a VP, SVP, or C-level title in R&D, Strategy, or Business Development at a mid-cap or large pharma. Their day is dominated by budget reviews, pipeline portfolio sessions, and one-on-ones with department heads. They are evaluated on capital efficiency — how much pipeline progress they can deliver per dollar of R&D spend — and the conversation in their head when they see a new tool is always "what does this displace, and how soon does it pay back."

What they fear: line-item additions that do not have an offsetting reduction somewhere; tools that require a long internal rollout before showing value; vendors who cannot tell them in a single slide what the ROI calculation looks like. What they trust: quantified case studies from peer companies, named references they can call, and a vendor who arrives with the business case half-built. They also trust their own purchasing team — and that team responds to compliance language, security framing, and predictable enterprise pricing.

Why this variant exists

The production page does the math implicitly: scale (1.2B sequences), accuracy (99.2%), and speed (50% faster) imply a value story but never explicitly compute it. For an executive buyer, implicit math is unconvincing — they have heard every vendor claim accelerated decision-making. Version D makes the math explicit, leading with a $2.4M average annual savings number paired with two supporting metrics (62% reduction in patent analysis time, 40% increase in pipeline throughput).

The variant also restructures the proof. Instead of feature cards followed by case studies, Version D leads with the ROI levers (faster decisions, lower external spend, de-risked programs, higher throughput) and follows with three concrete customer outcome blocks that show exact dollar amounts and time savings. This is the structure of an executive briefing, not a feature tour.

User scenarios

Scenario 1 — A VP of R&D at a mid-cap pharma is in a Tuesday morning portfolio review where the CFO has questioned why the IP team's external counsel spend grew 18% year-over-year. He remembers an Eureka case study he saw in his executive newsletter and pulls it up. The case shows a top-10 pharma reducing external counsel spend by 62% — the exact problem he is being asked about. He forwards Version D to his head of IP with "let's talk to these people."
Scenario 2 — A new Chief Scientific Officer is six weeks into the role and is being asked to deliver a 90-day plan that includes "tooling modernization." She lands on Version D from a sales rep's email. The four ROI levers map to four bullet points in her plan; the case studies give her concrete numbers to use in the board deck. She schedules an ROI assessment call within an hour.
Scenario 3 — A BD lead at a specialty pharma is preparing a budget request to expand the BD team. She wants to argue for a tooling investment instead of a headcount investment. The "doubled deal pipeline in 6 months" case in Version D is exactly the proof point she needs. She sends the page to her VP with the budget request as supporting evidence.

Content rationale

The hero composition is deliberately asymmetric. The left column carries the message ("Drive pharma R&D ROI") and three executive-friendly stats (3.2× ROI, 62% spend cut, 100+ enterprise clients). The right column carries a primary ROI panel ($2.4M average annual savings) flanked by two smaller metrics. The mono numerals at large size are a deliberate echo of how P&L numbers appear in financial presentations — the executive reader recognizes the typographic register even before they parse the content.

The four ROI lever cards (faster decisions, lower external spend, de-risked programs, higher throughput) are framed as business outcomes rather than product features. Each card carries a quantified claim ("Lead optimization cycles cut from 6 weeks to 5 days"; "60%+ savings typical"; "$50M+ cost of a late-stage surprise"). These numbers should be defensible — if they cannot be backed up by case data, they should be softened, because the persona will check.

The customer outcome row uses three named-but-anonymized companies (Top-10 Pharma, Mid-Cap Biotech, Specialty Pharma) with quantified outcomes in mono-formatted success-green metrics. The success green color is a deliberate semantic choice from the design system — for a numerical outcome, green is the color of "this worked," and the variant earns the right to use it because every number is a real outcome.

The CTA is "Schedule ROI assessment" rather than "Book a demo" or "Start free trial." This is an aggressive specialization that says to the reader: we are not selling features, we are offering to model your specific business case. It is a higher-friction CTA than self-serve trials, but it is the right one for this persona because executive buyers do not start free trials — they delegate procurement.

Where to deploy

Paid: targeted display ads on Endpoints+, BioPharma Dive's executive newsletters, and STAT Plus. Outbound: sales rep follow-up email (the page works well as a "as discussed" link). Account-based marketing: retargeting visitors from named accounts who have already engaged with product content. Conference: CFO and CSO tracks at JPM Healthcare, BIO, and DCAT — Version D printed onto a one-pager works equally well as a leave-behind.

Where it underperforms

Version D is a poor match for technical evaluators. A medicinal chemist landing on Version D will not see SAR or ADMET in the hero and will assume the tool is a strategy product, not a research product. Route technical evaluators to Version A or F.

It is also a wrong choice for early-stage biotech buyers, who will read the enterprise framing as "not for us." The $2.4M annual savings number, while real, is not relatable to a 30-person company whose entire opex is $10M. Route those buyers to Version B.

Finally, Version D will underperform with users who arrive without prior context — the persona this page is designed for already knows what Eureka does and is evaluating the business case. A cold visitor will find the variant skips too quickly to outcomes and miss the product story. Route cold traffic to the production page or to Version E.

E
Version E — Minimal
Decision-ready buyer · transparent pricing · no fluff
Minimal Self-serve

Persona portrait

The reader has already done their research. They came to the page from a branded search ("eureka patsnap pricing"), a procurement portal, a peer recommendation, or a previous trial that lapsed. They know what Eureka does in broad strokes and they are not looking to be educated — they are looking to confirm pricing, scan the feature set one more time, and make a decision. This is a high-intent visitor at a low-friction moment, and any extra section between them and the price card costs conversion.

SECTION 06

Recommended A/B test design

Having ten variants does not mean running a ten-way test. The most common failure mode for this kind of work is to ship all variants behind a flat traffic split, wait for "winners," and conclude nothing. This section explains why that approach fails and offers two designs that work.

Why a flat ten-way split fails

Two reasons. First, statistical power: splitting traffic ten ways means each variant receives one-tenth of the volume that a two-way A/B would receive. Reaching significance takes ten times longer; for any but the highest-traffic pages, this means the test never concludes within the planning horizon. Second, confounded variables: each variant differs from production in multiple dimensions (hero composition, section order, CTA copy, content emphasis). When B beats production, the lift could be attributable to any of those dimensions — there is no way to know which change drove the difference.

The honest version of a flat ten-way split is "we will pick whichever variant has the highest conversion at the end of the quarter and ship it." This is fine if the goal is incremental optimization. It is wrong if the goal is to learn what the audience actually wants — which is the implied goal of building ten variants in the first place.

Researcher note — if leadership presses for a flat ten-way split, the right counter is to compute the required sample size for a 5% lift detection at 80% power. Most internal-tool landing pages will need months of traffic at 10% per variant. Show the math; the conclusion sells itself.

Approach 1 — Persona-routed (recommended)

Bucket users into segments based on traffic source before they hit the page, then run small A/B tests inside each bucket. Each bucket gets a primary variant (matched to the segment from section 4) and the production page as control. Two variants per segment is enough to learn whether specialization beats consensus for that segment.

Routing logic should live at the redirect layer or in the Cloudflare worker, not inside the page itself. Examples:

  • Visitors from LinkedIn campaigns targeting "Director of Discovery" titles → 50% Version B, 50% production
  • Visitors from technical Slack or RDKit lists → 50% Version A, 50% production
  • Visitors from named-account ABM campaigns → 50% Version G, 50% production
  • Visitors from branded search ("eureka pricing") → 50% Version E, 50% production
  • Visitors from Hacker News, dev.to, or GitHub README links → 50% Version F, 50% production
  • Visitors from QR codes or mobile-detected devices → 50% Version I, 50% production

The advantage of this approach is that each test is interpretable. If Version B beats production for LinkedIn-sourced biotech leads but Version F beats production for HN-sourced builders, both lifts are real and additive — and both can ship to production-segment routing without reverting the unified page.

The disadvantage is that the test only runs on segmented traffic, so untaggable visitors continue to see the production page. This is acceptable: the production page is a consensus message and the test does not need to address everyone.

Approach 2 — Axis-isolated

Pick two or three variants that move only one variable versus production, and run a clean three-way test. The goal here is causality: knowing which dimension drives lift, not just which page wins.

Suggested first-wave isolated tests:

  • Test 1 — Specificity vs Generality. Production vs Version A vs Version B. Same audience (existing organic search traffic), different specialization angles (technical vs startup-energetic). Question answered: does specialization in either direction beat the consensus message for general traffic?
  • Test 2 — Proof type. Version C (peer-reviewed) vs Version G (case-driven) vs production. Same late-funnel audience (visitors with two or more prior page views), different proof structures. Question answered: which evidence type converts late-funnel visitors faster?
  • Test 3 — Pricing transparency. Production vs Version E. Same branded-search audience. Question answered: does putting pricing above the fold raise self-serve conversion?

The advantage of axis-isolated testing is causal clarity. The disadvantage is that it leaves several variants un-tested in the first wave (D, F, H, I, J). The right resolution is to run Approach 1 in parallel for variants whose audience is clearly segmentable, and Approach 2 for variants whose audience overlaps with production traffic.

What to ship first

If only one test could ship in week one, run Test 3 above (production vs E for branded search). Branded-search traffic is the highest-intent segment; the change is small (one page) and reversible; the metric is unambiguous (self-serve Pro signups). A win here is fast, defensible, and immediately actionable.

If two tests could ship in week one, add the persona-routed test for LinkedIn biotech traffic (production vs Version B). LinkedIn is segmentable and high-volume; B is the variant with the most specialized message-market fit; and the metric (trial signups within 7 days) is clean.

Each variant should be measured against a primary metric appropriate to its persona, not a single shared conversion event. Section 4 listed these per row; the consolidated framework is:

Guardrail metrics

Every variant should also be tracked against two guardrails: bounce rate (signals when the variant has been served to the wrong audience) and scroll depth (signals when the structure is wrong for the audience even if they don't bounce). A variant that wins on conversion but spikes bounce rate is being served to traffic it should not see; the lesson is to tighten the routing, not to revert the variant.

How long to run

For each test, plan to reach the lesser of (a) statistical significance at 80% power for a 10% lift, or (b) two full marketing-cadence cycles (typically eight weeks). Running shorter risks day-of-week and campaign-driven noise; running longer turns the test into the new production page without explicit decision.
